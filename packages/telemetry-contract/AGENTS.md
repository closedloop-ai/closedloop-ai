# Telemetry Contract Guidelines

- Keep OTel-owned attribute groups separate from ClosedLoop compatibility attributes. When adding or moving attributes, update ownership maps and tests that prove each group matches its declared ownership.
- Keep Zod validators, generated JSON schema patterns, and schema-check payloads behaviorally aligned. Edge cases added to one representation must be covered in the other representation's validation path.
- Shared telemetry payload fixtures used by both tests and package scripts belong in a shared test-fixture module owned by this package, not duplicated in each caller.
- Do not use Vitest source scans to enforce where contract literals live. Assert exported constants or runtime behavior in unit tests, and use a script or lint-style check for source-layout invariants.
- Validate JSON parsed from external commands or APIs with a schema or narrowing helper before iterating over it or returning typed contract state.
- Package artifacts and shipped samples are scanned for local absolute paths. Do not embed local temp, home, workspace, or drive-root path literals in files that can be packed; construct runtime temp defaults without source literals and keep package-content coverage when adding publishable files.
- Tests for stalled sample-script network paths must bound the external command's own timeout and force-close harness server connections during cleanup so CI latency does not depend on socket defaults.
- Package-local export paths, export targets, and Bash constants referenced by multiple tests or scripts must be imported from a package-owned constant module or parsed/asserted from the source owner instead of duplicated as magic values.
- The collector allow-list (`collector/allowed-attributes.json` and `collector/keyless-telemetry-redaction.yaml`) is generated from `TelemetryAttribute` via `CollectorAllowedAttributeKeys` (exported from `src/attributes.ts`); never hand-edit it. When the attribute value set changes, run `generate:collector-allowlist` and commit the regenerated artifacts in the same PR — `check:collector-allowlist` (in `build`) fails on drift. These artifacts are committed at the package root and are intentionally not packed (`files: ["dist"]`) nor exported, so they do not touch the release-asset set.
- The collector PostHog-routing fragments (`collector/posthog-product-signals.json`, `collector/keyless-telemetry-posthog-routing.yaml`, `collector/posthog-identity-transform.json`, and `collector/keyless-telemetry-posthog-identity.yaml`) are generated by `generate:collector-posthog-routing` from `CollectorProductSignalAttributeKeys` (exported from `src/attributes.ts`) and the generator's PostHog identity source/target/processor mapping (FEA-1991/FEA-2184); never hand-edit them. When the product-signal marker set or identity mapping changes, run `generate:collector-posthog-routing` and commit the regenerated artifacts in the same PR — `check:collector-posthog-routing` (in `build`) fails on drift. Shared codegen + drift-guard helpers (YAML-safety, artifact writer, parsing predicates, `readCommittedArtifact`/`repoRootAnnotationPath`) live in `scripts/collector-codegen-common.ts`. Same packing/export exclusion as the allow-list artifacts.
- Adding or removing a publishable contract surface (a new `./<name>` entry in `package.json` `exports`, or a new generated schema) changes the release-asset set. Update every synchronized list in the same change: `package.json` `exports`, `ExpectedTelemetryContractReleaseAssetNames` (`scripts/deploy/publish-telemetry-contract-preflight.ts`), the publish workflow `on.push.paths` (`.github/workflows/publish-telemetry-contract.yml`), and the workflow-source test path filters (`scripts/deploy/publish-telemetry-contract-workflow-source.test.ts`). The `deploy-scripts` CI job is gated on `package.json` and the publish workflow, so the consistency guard that proves these lists agree runs on the PR that introduces the surface — not downstream.
