# Desktop App Guidelines

## Ownership

`apps/desktop` is the source-owned Closedloop Desktop/Electron client inside the `symphony-alpha` monorepo. Do not treat `closedloop-ai/closedloop-electron` as an active implementation or release owner; references to that repository must be explicitly historical.

## Versioning And Build Info

> **â›” Do NOT edit the `version` field in `apps/desktop/package.json` as part of your change.** The patch number is bumped automatically on merge to `main` (`desktop-version-bump.yml`) whenever the `apps/desktop` build closure changed â€” you do not, and must not, hand-bump it per-PR. If your diff touched `apps/desktop`, the release counter advances on its own; adding a manual patch bump only conflicts with the ledger. Only change this field to deliberately raise the **floor** or advance the `major.minor` **series**, and only when the task explicitly asks for it. Revert any incidental patch bump before opening a PR.

`apps/desktop/package.json` declares the `major.minor` release **series anchor** plus a **patch floor** (e.g. `0.16.71`); the live counter lives in the `desktop-v*` git tag / GitHub Release ledger. On every merge to `main`, `desktop-version-bump.yml` mints the next `desktop-v<patch>` **tag** â€” but only when the `apps/desktop` build closure (the app plus the `@repo/*` packages it bundles) changed, detected via turbo's `[before...sha]` commit-range filter (FEA-2135). The next version is `max(highest ledger patch in the series + 1, manifest patch)` (FEA-2156), so the manifest patch is a **floor**: raise it to deliberately jump the next release (e.g. above a high-water mark the ledger has not reached), but do NOT hand-bump it per-PR â€” once the ledger passes the floor, it leads on its own. **Release Desktop** builds and publishes the commit the highest reachable `desktop-v*` tag points at (`scripts/deploy/read-desktop-version.ts`), so released numbers may skip (we don't ship every commit). Advance the series the same way (e.g. `0.16.x` â†’ `0.17.0`). `src/shared/build-info.ts` is generated by `pnpm -C apps/desktop prebuild` and must not be committed.

Run `pnpm -C apps/desktop prebuild` before Desktop lint, typecheck, source assertions, or tests that load app entrypoints. Main process output is JS-only under `dist/`; do not commit generated `dist`, `.generated`, package-stage, or DMG output.

## Local Database Schema (Prisma)

> ### âš ď¸Ź The local DB engine is SQLite (libSQL) â€” NOT Postgres/PGlite
>
> FEA-2038 migrated the desktop local store from **PGlite** (Postgres-in-WASM) to **SQLite via libSQL**. The runtime file is `src/main/database/sqlite.ts` (formerly `pglite.ts`); its symbols now use the `sqlite`/`Sqlite` prefix (`openSqliteAgentDatabase`, `SqliteExecutor`, â€¦) â€” **the engine is SQLite.** The on-disk store is `agent-dashboard.sqlite` under Electron `userData`. Prose further down in this section that still says "PGlite", "Postgres", `pnpm db:dev`, or `agent-dashboard.pgdata` is **stale and pending rewrite** â€” trust this callout over it.
>
> **All hand-written SQL â€” raw reads/writes, aggregates, AND migration `.sql` files â€” MUST be SQLite dialect.** Postgres syntax silently passes typecheck/lint and then throws at runtime against the live store. Forbidden Postgres constructs and their SQLite replacements:
>
> | âťŚ Postgres (do NOT use) | âś… SQLite (libSQL) |
> | --- | --- |
> | `expr::int` / `expr::text` / `expr::timestamptz` casts | `CAST(expr AS INTEGER/TEXT)` or just omit; `Number()`-coerce counts in JS |
> | `ILIKE '%x%'` | `lower(col) LIKE '%x%'` (SQLite `LIKE` is already ASCII case-insensitive) |
> | `col ~ '^[0-9]{4}-...'` (regex) | `col GLOB '[0-9][0-9][0-9][0-9]-[0-9][0-9]-[0-9][0-9]*'` |
> | `EXTRACT(EPOCH FROM (a - b)) * 1000` | `(julianday(a) - julianday(b)) * 86400000` |
> | `COUNT(*) FILTER (WHERE p)` | `SUM(CASE WHEN p THEN 1 ELSE 0 END)` (or `COUNT(*) FILTER` â€” libSQL supports it, but the codebase prefers the `SUM(CASE â€¦)` idiom) |
> | `DO $$ â€¦ to_regclass(â€¦) â€¦ $$` guards in migrations | plain `CREATE INDEX IF NOT EXISTS â€¦ ON â€¦(â€¦) WHERE â€¦;` (SQLite supports `IF NOT EXISTS` + partial indexes natively) |
> | Dates: comparing `started_at::timestamptz` | compare ISO-8601 **strings** directly (the app only ever persists `toISOString()`); guard malformed values with the `GLOB` date-prefix expr |
>
> **Canonical pattern to mirror** (already SQLite-correct, verified against the live store): `buildUsageFilterClause`, `aggregateSqliteUsage`, and the `SESSION_STARTED_AT_TS_EXPR` / `SESSION_STARTED_AT_BOUNDS_EXPR` date guards in `sqlite.ts`. Before finishing any SQL change, grep your diff for `ILIKE`, `::`, `EXTRACT`, `~ '`, `to_regclass`, `DO $$`, `timestamptz` â€” there must be **zero** matches in new SQL.
>
> **Analytics invariant (FEA-2038):** dashboard analytics/usage/branch reads must compute **metadata-only SQL aggregates** (`COUNT`/`SUM`/`GROUP BY` over the metadata/token tables) and must **never hydrate the full session/event corpus into JS** â€” doing so OOMs the db-host utilityProcess (exit code 5) on a large local history. Row-loading list reads default to a 90-day window (all-time via an explicit toggle); aggregates stay all-time (cheap in SQL).
>
> **Migrations** are sequential `000N_<name>/migration.sql`, hand-authored in SQLite dialect (see `0001_init`, `0002_idx_token_events_session_created`, `0003_idx_branch_name`). Use the next unused contiguous prefix. The already-merged FEA-2030 timestamped migration is immutable and grandfathered by the manifest order; do not add more timestamp-prefixed migrations. Regenerate the gitignored manifest with `node apps/desktop/scripts/generate-migrations-manifest.mjs` after adding one. (This supersedes the timestamp-prefix and Prisma-Postgres-generation guidance below.)

The local store schema is owned by `apps/desktop/prisma/schema.prisma` (independent of the cloud schema in `packages/database` by design â€” FEA-1736 decision 6). The generated client output `src/main/database/generated/` is gitignored; `pnpm db:generate` produces it and runs automatically inside `build`, `typecheck`, and `test`.

Schema changes are forward-only Prisma migrations applied at boot by the runtime migration runner (`src/main/database/migration-runner.ts`). To change the schema: edit `prisma/schema.prisma`, generate a migration (`pnpm db:migrate:dev` against a throwaway Postgres via `pnpm db:dev`), and commit it under `prisma/migrations/`. The runner records applied migrations in a `_desktop_migrations` tracking table, applies each pending migration in its own transaction (no down-migrations), and refuses to boot on checksum drift or a downgraded app â€” surfaced as an Agent Monitor boot failure with DB IPC disabled, never a crash loop. A prebuild step (`scripts/generate-migrations-manifest.mjs`) embeds the migrations into the gitignored `src/main/database/migrations-manifest.ts` so they survive asar packaging (mirrors `build-info.ts`).

**A migration is immutable once it merges to `main`.** Editing, deleting, or renaming a committed `migration.sql` changes its embedded checksum, so the runner refuses to boot for any store that already applied the old bytes (the exact checksum-drift refusal above). Never edit a merged migration; add a NEW forward migration instead. CI enforces this for both Prisma targets (cloud `packages/database` and desktop `apps/desktop`) via the `migration-immutability` job in `.github/workflows/pr-test.yml`, gated on the `database` and `desktop_database` path filters. If a local dev store hits a checksum-drift refusal after pulling `main` (because it applied an interim version of a migration that was corrected before any desktop release shipped it), reset the local store to recover: quit the app and delete the `agent-dashboard.sqlite*` files under Electron's `userData` path (macOS: `~/Library/Application Support/Closedloop/agent-dashboard.sqlite*`, including the `-wal`/`-shm` sidecars); it rebuilds from scratch on next boot. Only do this for unreleased migration churn; a released migration is permanent and must be fixed forward, never by editing it.

New desktop migration directory names must use the next contiguous `000N_` prefix in `apps/desktop/prisma/migrations/`. Do not rename the existing timestamp-prefixed FEA-2030 migration; it has a pinned manifest sort key because merged migrations are immutable. Do not insert a pending migration before an already-applied migration: changing manifest order can create a `HistoryGap` refusal on upgraded installs.

`src/main/database/baseline-schema.ts` holds the FROZEN legacy DDL snapshot as of the runner cutover plus the `BASELINE_MIGRATIONS` it represents. **Do not edit it or append to `BASELINE_MIGRATIONS`** â€” it exists only to baseline a pre-runner install (re-assert the idempotent legacy DDL once, then record the baseline migrations as applied *without executing* them). Post-cutover schema lives only in new migrations.

The migration-refusal contract (`MigrationRefusalKind`, `DesktopMigrationError`, `userFacingMigrationRefusal`) lives in `src/main/migration-refusal.ts`, **deliberately outside `src/main/database/`**. Boot files (`app.ts` surfaces a refusal as an Agent Monitor boot failure) cannot statically import the database runtime â€” the boundary guard (`scripts/dependency-cruiser.config.cjs`) forbids any boot-graph value import into `src/main/database/**` (which transitively pulls in SQLite). Keeping the pure, DB-free contract in its own module lets the runner (producer) and the boot handler (consumer) share it without dragging the DB runtime into the boot bundle.

### Typed Prisma client

`src/main/database/prisma-client.ts` (`createDesktopPrisma`) builds the desktop store's ONLY access layer over the libSQL `config`, created inside `openSqliteAgentDatabase` and exposed as `agentDatabase.prisma`. It owns the connection topology: ONE **writer** `PrismaClient` plus a small pool of `query_only` **reader** `PrismaClient`s (default 2, `connection-pragmas.ts` is the single source of truth for the WAL/busy-timeout/cache PRAGMAs every connection applies). Three verbs, no second layer to choose between:
> - **`prisma.write(fn)`** â€” all writes + `$transaction`, serialized through the shared write queue on the writer connection. A throttled `wal_checkpoint(TRUNCATE)` fires after each write to bound WAL/RSS during the backfill.
> - **`prisma.read(fn)`** â€” heavy/independent reads that must run CONCURRENTLY with the first-launch backfill writer (sync hydration, analytics/usage aggregation, cursor paging). Dispatched round-robin to a reader connection; each reader self-serializes its own statements via the adapter's per-connection `async-mutex` (the SIGTRAP guard â€” one connection never runs two statements at once), and separate readers run in parallel over the WAL snapshot. `fn` gets a read-only client with a read-scoped `$transaction` for multi-read snapshots.
> - **`prisma.client`** â€” light reads co-located with writes (read-your-writes) on the PRIMARY/writer connection.
>
> `prisma.client` and the `prisma.read` client are both typed as a **read-only facade** (`DesktopPrismaReader`) â€” calling a mutation method is a compile error, so the write-queue rule is type-enforced, not convention; `query_only=ON` backstops the reader pool at the engine (a write faults with SQLITE_READONLY). The community adapter is exact-version-pinned and confined to this one module so a swap to the FEA-1736 in-house `SqlDriverAdapter` fallback touches one file â€” **production code must not import the adapter anywhere else**. The sole exception is the characterization test (`test/prisma-sqlite-adapter.test.ts`), which imports the adapter directly on purpose to exercise the raw clientâ†”adapter contract against the real schema; all other tests go through the factory (`test/prisma-test-utils.ts`).

There is ONE client â€” no separate raw `storeDb` handle. Queries that don't map cleanly to a typed delegate run on the same `DesktopPrisma` client via Prisma's raw escape hatch (`$queryRawUnsafe` for reads, `$executeRawUnsafe` for writes). The community adapter implements `queryRaw`/`executeRaw`, so raw SQL works through it (characterized in `test/prisma-sqlite-adapter.test.ts`: `$queryRaw`/`$transaction` and the `$queryRawUnsafe`/`$executeRawUnsafe` positional-param variants the write path uses). Raw **writes** are reachable only inside `prisma.write(...)` (the full client); raw **reads** outside a write use `$queryRaw`/`$queryRawUnsafe`, which the `DesktopPrismaReader` facade re-exposes read-only (`$executeRaw*`/`$transaction`/typed mutations stay compile-errors on the facade). The read-vs-write split on `$queryRaw*` is by convention â€” SQL can't be statically classified â€” so never smuggle a mutation through it; raw writes go through `prisma.write`.

**Typed-first is the default.** Every store runs on the single `DesktopPrisma` client: typed delegates where the query has a clean form, and the raw `$queryRawUnsafe`/`$executeRawUnsafe` escape hatch (on the same client) only for SQL with no clean typed form â€” aggregations and window/recursive-CTE queries, group-bys over a joined relation's columns, `@@ignore`'d tables (e.g. `token_events`, which has no PK and so no generated delegate), and hand-tuned `INSERT â€¦ ON CONFLICT DO UPDATE` upserts whose COALESCE/CASE merge logic Prisma's `upsert` can't express. The raw hatch is for un-typeable SQL, NOT a shortcut. One nuance worth keeping: a single server-side aggregate-join (e.g. the session-detail `COUNT(agents)`/`COUNT(events)`/`SUM(tokens)` fold) stays raw on purpose â€” replacing it with per-table `groupBy` reads marshalled into a JS join is a real performance regression, so performance is a valid reason to keep raw SQL. INTEGER/`BIGINT` columns that a raw read can surface as `bigint` through the adapter are coerced to JS numbers at the existing `Number()` / `readStorageTokenCount` / `token()` boundaries (a bare `BIGINT` column surfaces as `bigint`; a `SUM`/`COUNT`/`COALESCE` expression stays `number`). **Packaging:** the `prisma-client` generator emits pure TypeScript that `tsc` compiles into `dist/` like any other module, and `@prisma/client`'s driver-adapter runtime ships no `.wasm` here â€” so `stage-packaging-app.mjs`'s recursive `dist/` copy carries the client with no extra asset-copy step.

Guard tests: `test/prisma-baseline-equivalence.test.ts` (frozen baseline â‰ˇ the `BASELINE_MIGRATIONS` chain), `test/prisma-migrations-agreement.test.ts` (migrations â‰ˇ `schema.prisma`), `test/migration-runner.test.ts` (fresh / baseline / older-legacy / pending / rollback / drift / downgrade), `test/prisma-sqlite-adapter.test.ts` (the client over the adapter against the real schema), `test/prisma-client-factory.test.ts` (factory reads + `write()` queue serialization + the reader-pool concurrency / `query_only` proofs), and `test/pack-catalog-seed.test.ts` (the catalog seed: JSON round-trip, seed-version gating, live-field preservation, and that the bundled `catalog-seed.json` populates the visible catalog via `listCatalog` â€” guarding the empty-Packs boot regression). ### DB host process boundary (FEA-2038)

SQLite + Prisma + every store run in a dedicated **DB host** `utilityProcess` (`database/db-host/`), NOT the main process. The main process holds `agentDatabase` as a **forwarding proxy** (`createDbHostAgentDatabase`) that turns each `agentDatabase.<path>(args)` call into one IPC `invoke` to the child. It works for any method whose **args and result are structured-clone-safe**: plain objects, arrays, numbers, strings, `null`. `prisma.client.*` reads work (delegate args are plain objects).

> **Never call a callback-taking method on the proxy from the main process.** `agentDatabase.prisma.read(fn)` / `agentDatabase.prisma.write(fn)` pass a **function**, which cannot be structured-cloned across the process boundary, so the call rejects with `DataCloneError: "An object could not be cloned"`. The `prisma.read`/`prisma.write` verbs exist to run a callback AGAINST the live connection, so they are only callable INSIDE the DB host, never from main through the proxy.

When main needs data or a write that internally uses `prisma.read(fn)`/`prisma.write(fn)`, do NOT reach for `agentDatabase.prisma` from main. Instead expose it as a **clone-safe op that runs in the child** and call that:
> - **A method on `SqliteAgentDatabase`** (no callback args, serializable return), e.g. `sessions.count()`, `triggerEnrichmentSweep()`, `remediateMisattributedPrBranches()`. The proxy forwards the dotted path and the real method runs its `prisma.read/write` callback locally in the child. This is the default.
> - **A `store:`-prefixed op** registered in `db-host-worker.ts` `storeOps` (e.g. `catalog.fetch.run`) for callback-taking store functions that live outside the `SqliteAgentDatabase` surface, invoked from main via `invokeStoreOp(...)`.

The proxy is deliberately loud: a missed reroute throws `DataCloneError` at the call site rather than silently returning wrong data. "An object could not be cloned" in the desktop logs means a main-process caller is passing a callback (or other non-cloneable value) through the proxy. Reroute it to a child op. Regression-guarded by `test/agent-dashboard-ipc-perf.test.ts` (the `session_count` dimension counts via the clone-safe `sessions.count()` method, never `prisma.read` from main).

## Electron And Gateway Rules

Gateway operations live in `apps/desktop/src/server/operations/` because they need local filesystem and process access. Do not reimplement them in `apps/app` or `apps/api`.

Binary path discovery is centralized in `src/server/shell-path.ts`. Use `getShellPath()` / `getShellPathSync()` and `resolveBinaryFromLoginShell()` / `resolveBinaryFromLoginShellSync()`; do not add direct `which` calls or shell reimplementations.

When Desktop code uses a resolved shell PATH to decide whether a CLI or harness is available, pass that same resolved PATH into the child process env used for the selected command, or document and test the intentional difference.

Gateway auth must fail closed. Missing Desktop API keys may make local browser gateway mode unavailable with an explicit error, but must not crash app startup, silently fall back to insecure auth, or expose gateway routes by origin alone.

Guard every `webContents.send()` call with `!window.isDestroyed()` and `!window.webContents.isDestroyed()` â€” Electron objects can throw after sleep/wake or frame disposal. Use `sendToRendererWindow` from `src/main/renderer-ipc.ts` (or `DesktopWindow.sendToRenderer()` when you have the window object).

Sandbox access is owned by `isPathAllowed()` in `src/server/security.ts` and `buildAllowedDirectories()` in `src/shared/sandbox-policy.ts`. Sensitive directories remain denied even if a parent path is allowed.

### `gitBranch` is not attribution data (FEA-2531)

Harness-reported git branch fields â€” `session.gitBranch` (captured once at session start) and per-line `NormalizedToolUse.gitBranch` â€” report the branch of the session's CWD checkout. They are wrong for **every** worktree flow (the session sits in the main checkout on `main` while all real work happens through `cd ../worktree && â€¦`) and go stale after any mid-session checkout. Do not use them to attribute branches, PRs, or cost. Attribution must derive from `session_artifact_links` relationships produced by the artifact-ref extractor: write methods (`git_push` / `gh_pr_create` / `git_commit`) carry `relation: "created"`; reads (`start_branch` / `git_checkout` / `git_worktree_add`) stay `workspace` and never attribute. A created PR's head ref resolves evidence-first (gh's own create output / `--head` flag, else the session's nearest preceding branch write); `tu.gitBranch` is the last-resort fallback only, and default branches are always rejected (FEA-2260). When touching branch/PR attribution, trace the change through the extractor dedup, `createdPrHeadBranches`, the `pull_requests` upsert, and the link-propagation passes â€” not just the emission site.

Harness discrimination must stay closed-set. Match known service or harness identifiers exactly, or by documented namespace prefixes, instead of substring scans over arbitrary metric names, span names, attribute keys, or attribute values. Add a negative test for substring-only false positives when adding a new identifier family.

Optional capture/runtime modules must not crash their owner at import time because a descriptor, parser, or dependency-heavy runtime initialized at module scope failed. Initialize those resources lazily or behind the runtime start/request path, surface failures through the existing unavailable/log path, and keep raw payloads out of diagnostics.

Async runtime startup continuations must not restart dependent subsystems after shutdown or supersession. When a `start()` path awaits or attaches a callback before starting collectors, listeners, or sidecars, capture the current start promise or generation and check both the closed flag and that captured identity before mutating runtime state.

Local session source readiness is owned by the SQLite-backed renderer IPC source, not optional capture sidecars. Hook listener, collector, or OTLP startup failures may degrade capture, but they must not make Desktop Sessions unreadable once DB IPC/runtime readiness is established; cover mixed ready-plus-capture-failed states when changing this lifecycle.

In `stop()`/`destroy()` methods, clear all accumulated state (Maps, Sets, timestamps, intervals) â€” stale values from a previous lifecycle corrupt analytics or behavior on restart.

Distinguish transient failures (network timeout, CLI not found in PATH, gh auth expired) from permanent impossibility (no git directory, file format unsupported). Only set `not_applicable` or permanent failure states for the latter; transient failures should leave retry eligibility intact.

### Collection-mode mutual exclusivity (FEA-1839)

A harness's live telemetry is captured by exactly one channel at a time: the Closedloop hook handler (`"hooks"`) or the live JSONL file watcher (`"watcher"`) â€” never both. Running both for one harness double-counts every tool-call and lifecycle row. `getActiveCollectionMode(harness, hooks)` in `src/main/collectors/collection-mode.ts` is the single source of truth for this routing; `CollectorManager` starts a live watcher only for `"watcher"`-mode harnesses (`"hooks"`/`"disabled"` run the idempotent boot import once, no watcher). Every new harness must derive its live-collection decision from this function â€” do not add an inline hooks-installed conditional at a call site. The `mutual-exclusivity-monitor` records a single `mutual_exclusivity_violation` event row if both channels ever emit the same harness session in one process lifetime.

## Idempotent Re-Processing: Revision Constants

Desktop uses three independent version constants to trigger idempotent re-derivation of local data. Each governs a different concern â€” bumping one does NOT affect the others.

### DATA_REVISION (`src/main/collectors/data-revision.ts`)

**Current value:** 3. Controls session-level re-derivation from source transcripts.

Every session row is stamped with `data_revision = DATA_REVISION` at import time. On boot, `data-revision-rebuild.ts` queries `SELECT ... FROM sessions WHERE data_revision != <current>`, re-parses each stale session from its source transcript, and transactionally replaces derived rows (events, agents, token_usage, session_artifact_links).

**When to bump:** Parser or import semantics changed in a way that should re-derive already-imported sessions. Examples: event ID generation changed (rev 3), dedup/merge logic changed (rev 2), new fields extracted from transcripts.

**How:** Increment the number in `data-revision.ts`, add a changelog entry to the revision history comment. Sessions with the old value are rebuilt on next app launch; active sessions heal via their normal reimport path. Sessions whose source transcript is gone are left untouched.

**Does NOT trigger:** enrichment re-run or catchup cache invalidation.

### EXTRACTOR_VERSION (`src/main/collectors/parsing/artifact-ref-extractor.ts`)

**Current value:** 8. Controls artifact-reference re-extraction from parsed sessions.

The artifact-ref-extractor runs at parse/import time and produces `session_artifact_links` rows. Each link row is stamped with `extractor_version`. The backfill pass (`artifact-link-backfill.ts`) queries for sessions whose link rows carry a stale extractor version and re-runs the extractor.

**When to bump:** Extraction semantics changed â€” new artifact kinds detected, relation classification changed (e.g. commit refs changed from `output` to `created`), new regex patterns added, confidence scoring changed.

**How:** Increment `EXTRACTOR_VERSION` in `artifact-ref-extractor.ts`. Existing installs will re-derive artifact links on next boot via the backfill pass (links are delete-then-reinsert, so refs the new version no longer emits drop out of every link-driven surface). Since FEA-2608 the local enrichment sweep is no longer triggered automatically; GitHub fields on new artifacts are filled by cloud hydration when GitHub is connected.

**Does NOT trigger:** session re-parse (that's DATA_REVISION) or catchup cache invalidation.

### ACTIVITY_CLASSIFIER_VERSION (`src/main/collectors/parsing/activity-segment-classifier.ts`)

**Current value:** 2. Controls activity-segment re-derivation (state-aware spend attribution, FEA-2267 / FEA-2269 / PRD-488). v2 (FEA-2269) replaced the FEA-2267 stub (one `other` segment per active span) with the real deterministic structural classifier â€” windowing + hysteresis + layered declaredâ†’structural scoring over the FEA-2268 evidence timeline â€” and extended the phase taxonomy from `{other, idle}` to the full active-work set (`explore`/`plan`/`implement`/`review`/`validate` + `other`/`idle`).

The classifier runs at parse/import time and produces `session_activity_segments` rows â€” the complete per-session activity-phase tiling. Each session's derivation is recorded in the `activity_segment_backfill_seen` high-water mark with `classifier_version`. The backfill pass (`activity-segment-backfill.ts`) queries for sessions whose marker carries a stale classifier version (or whose transcript mtime changed) and re-tiles them.

**When to bump:** Classification semantics changed â€” phase taxonomy added/renamed, idle-gap threshold changed, segment-boundary rules changed, evidence-layer derivation changed.

**How:** Increment `ACTIVITY_CLASSIFIER_VERSION` in `activity-segment-classifier.ts`. Existing installs re-derive all activity segments on next boot via the backfill pass.

**Does NOT trigger:** session re-parse (that's DATA_REVISION) or catchup cache invalidation.

### PERSIST_VERSION (`src/main/collectors/catchup-cache.ts`)

**Current value:** 2. Controls the file-level catchup cache (mtime/size fingerprints).

The catchup cache tracks which transcript files have been imported. When `PERSIST_VERSION` changes, the entire cache is invalidated, forcing a one-time full re-scan of all transcript files on next boot.

**When to bump:** Cache format or fingerprint semantics changed (e.g. new fields in the cache entry, different hashing). NOT needed for parse output changes (use DATA_REVISION for that).

**How:** Increment `PERSIST_VERSION` in `catchup-cache.ts`. Next boot discards the cache and re-scans all files.

### Enrichment State Machine (`src/main/enrichment/`)

Enrichment is pull-based, NOT revision-driven. Artifacts have an `enrichment_state` column with four values:

| State | Meaning |
|-------|---------|
| `null` | Never enriched. Sweep will attempt enrichment. |
| `provisional` | Enriched but may change (e.g. open PR, active branch). Sweep will re-check. |
| `final` | Enriched and stable (merged PR, validated commit). Sweep skips. |
| `not_applicable` | Cannot be enriched (no git dir, gh unavailable, max failed attempts). Sweep skips. |

**Triggers:** Three events cause an enrichment sweep: post-import, app startup, app focus. The sweep queries for artifacts with `enrichment_state IS NULL OR enrichment_state = 'provisional'` and processes them.

**To force re-enrichment of all artifacts:** Reset `enrichment_state` to `NULL` on the target rows:
```sql
UPDATE artifacts SET enrichment_state = NULL, enrichment_attempts = 0 WHERE ...;
```
The next sweep trigger will re-enrich them. There is no `ENRICHMENT_REVISION` constant â€” enrichment is stateless and re-derives from git/gh on each attempt.

**Constants:** `MAX_ENRICHMENT_ATTEMPTS = 5` (failed attempts before `not_applicable`), `ENRICHMENT_SWEEP_DEBOUNCE_MS = 5min`, `LEASE_STALE_MS = 5min`, `GH_RATE_LIMIT_INTERVAL_MS = 6s`.

### Summary: Which Constant To Bump

| Change | Bump |
|--------|------|
| Parser extracts different data from transcripts | `DATA_REVISION` |
| Artifact-ref extraction logic changed | `EXTRACTOR_VERSION` |
| Activity-segment classification logic changed | `ACTIVITY_CLASSIFIER_VERSION` |
| Catchup cache format/fingerprint changed | `PERSIST_VERSION` |
| Enrichment logic changed (git/gh queries) | Reset `enrichment_state` to NULL |

## Test Practices

Do not guard Desktop behavior by reading implementation source as text and regex/substring-matching it (`readSource`/`readFileSync` of a `src/**/*.ts(x)` file with `assert.match`/`.toContain`). Those guards break on every rename/reorder and can be satisfied by a comment. Use the right mechanism for the invariant:

- **Boundary / architecture invariants** â€” e.g. the boot graph (`src/main/index.ts`, `app.ts`, `window.ts`) must not statically import `src/main/database/**` or the design-system dashboard runtime (the SQLite/heavy-runtime boundary, see the Prisma section above), and the app-OTel runtime may only egress through the sanctioned relay transport â€” are enforced by **dependency-cruiser** (`scripts/dependency-cruiser.config.cjs`, run via `pnpm assert:design-system-boundary`), which resolves the real module graph (static value imports only) instead of scanning `app.ts` source text. Add or tighten a rule there rather than reintroducing a source-text scan. Note: the former `test/agent-dashboard-boundary.test.ts` additionally AST-checked structural invariants dependency-cruiser can't express â€” the `BrowserWindow` `webPreferences` lock (`contextIsolation: true`, no `nodeIntegration`, `sandbox: false`) and the preloadâ†”handler `desktop:db` channel-parity contract â€” and those checks were retired with it; they are **not** currently enforced by an automated guard. Reinstate either as a dedicated behavioral test (next bullet) if you need it, not by extending a deleted file.
- **Security / IPC gates** â€” allowlist enforcement before `shell.openExternal`, IPC handlers rejecting untrusted senders, disabled/live handler installation â€” are enforced by **behavioral tests** that invoke the handler/function and assert the effect. Follow `test/external-url-allowlist.test.ts` (`isAllowedExternalUrl` fail-closed policy), `test/desktop-session-manager.test.ts` (allowlist stubs `openExternal` and asserts it is not called), `test/ipc-profile-channels.test.ts` (a fake `handle` registrar + `isTrustedSender: () => false` proves handlers reject untrusted senders), and `test/agent-dashboard-disabled-ipc.test.ts` (disabled/branch responder installation). When an inline `app.ts`/`window.ts` handler cannot be reached without booting Electron, extract its registration into a testable function (mirroring `registerProfileConfigIpcHandlers`) rather than pinning it with a source regex.
- **Renderer wiring** â€” a component subscribes to an IPC event or invokes a `window.desktopApi` method â€” is enforced by a **vitest render test** (`src/renderer/**/__tests__/*.test.tsx`, run by `pnpm test:renderer`): mount the component with a stubbed `window.desktopApi`, dispatch the event / click the control, and assert the call. See `src/renderer/components/diagnostics/__tests__/logs-panel.test.tsx`, `src/renderer/components/__tests__/update-banner.test.tsx`, and `src/renderer/components/settings/__tests__/settings-panel-security-api-key.test.tsx`.

Reading declarative config/data (workflow YAML, `migration.sql`, JSON fixtures) is not source-text testing and remains allowed. See the root `AGENTS.md` "Test Practices" rules for the repo-wide statement.

## Validation

For Desktop changes, run the narrow Desktop command first while iterating, then include the relevant tests, typecheck, and lint coverage for the touched monorepo surfaces. Use Desktop build/package validation only when the change specifically touches Desktop build output, workspace package materialization, packaging paths, or release payload behavior.

### `test:node` determinism (FEA-2399)

The `test:node` slice must be deterministic â€” the `desktop` CI check gates on it, and a single flaky assertion reddens the whole job (the runner has no retry/quarantine). When a node test waits for async work to complete, **synchronize on a real completion signal** emitted by the code under test (e.g. resolve a `deferred()` from an injected callback/mock when the awaited effect lands), not on a bounded poll loop. If a poll loop is genuinely unavoidable, it MUST `throw` when its bound is exhausted â€” never fall through silently and let a later assertion see stale state. This is enforced pre-merge by `test/no-silent-poll-barrier.guard.test.ts`.

To reproduce non-determinism, stress-run a suite: `STRESS_ITERS=100 STRESS_FILE=test/ingest-orchestrator.test.ts pnpm --filter desktop test:node:stress` (exits non-zero on the first failing iteration). The `desktop-testnode-stress` nightly workflow runs this on the CI runner class and alerts on Slack.
