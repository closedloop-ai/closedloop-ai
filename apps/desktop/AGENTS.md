# Desktop App Guidelines

## Ownership

`apps/desktop` is the source-owned Closedloop Desktop/Electron client inside the `symphony-alpha` monorepo. Do not treat `closedloop-ai/closedloop-electron` as an active implementation or release owner; references to that repository must be explicitly historical.

## Versioning And Build Info

`apps/desktop/package.json` declares the `major.minor` release **series anchor** plus a **patch floor** (e.g. `0.16.71`); the live counter lives in the `desktop-v*` git tag / GitHub Release ledger. On every merge to `main`, `desktop-version-bump.yml` mints the next `desktop-v<patch>` **tag** — but only when the `apps/desktop` build closure (the app plus the `@repo/*` packages it bundles) changed, detected via turbo's `[before...sha]` commit-range filter (FEA-2135). The next version is `max(highest ledger patch in the series + 1, manifest patch)` (FEA-2156), so the manifest patch is a **floor**: raise it to deliberately jump the next release (e.g. above a high-water mark the ledger has not reached), but do NOT hand-bump it per-PR — once the ledger passes the floor, it leads on its own. **Release Desktop** builds and publishes the commit the highest reachable `desktop-v*` tag points at (`scripts/deploy/read-desktop-version.ts`), so released numbers may skip (we don't ship every commit). Advance the series the same way (e.g. `0.16.x` → `0.17.0`). `src/shared/build-info.ts` is generated by `pnpm -C apps/desktop prebuild` and must not be committed.

Run `pnpm -C apps/desktop prebuild` before Desktop lint, typecheck, source assertions, or tests that load app entrypoints. Main process output is JS-only under `dist/`; do not commit generated `dist`, `.generated`, package-stage, or DMG output.

## Local Database Schema (Prisma, FEA-1791)

> ### ⚠️ The local DB engine is SQLite (libSQL) — NOT Postgres/PGlite
>
> FEA-2038 migrated the desktop local store from **PGlite** (Postgres-in-WASM) to **SQLite via libSQL**. The runtime file is `src/main/database/sqlite.ts` (formerly `pglite.ts`); its symbols now use the `sqlite`/`Sqlite` prefix (`openSqliteAgentDatabase`, `SqliteExecutor`, …) — **the engine is SQLite.** The on-disk store is `agent-dashboard.sqlite` under Electron `userData`. Prose further down in this section that still says "PGlite", "Postgres", `pnpm db:dev`, or `agent-dashboard.pgdata` is **stale and pending rewrite** — trust this callout over it.
>
> **All hand-written SQL — raw reads/writes, aggregates, AND migration `.sql` files — MUST be SQLite dialect.** Postgres syntax silently passes typecheck/lint and then throws at runtime against the live store. Forbidden Postgres constructs and their SQLite replacements:
>
> | ❌ Postgres (do NOT use) | ✅ SQLite (libSQL) |
> | --- | --- |
> | `expr::int` / `expr::text` / `expr::timestamptz` casts | `CAST(expr AS INTEGER/TEXT)` or just omit; `Number()`-coerce counts in JS |
> | `ILIKE '%x%'` | `lower(col) LIKE '%x%'` (SQLite `LIKE` is already ASCII case-insensitive) |
> | `col ~ '^[0-9]{4}-...'` (regex) | `col GLOB '[0-9][0-9][0-9][0-9]-[0-9][0-9]-[0-9][0-9]*'` |
> | `EXTRACT(EPOCH FROM (a - b)) * 1000` | `(julianday(a) - julianday(b)) * 86400000` |
> | `COUNT(*) FILTER (WHERE p)` | `SUM(CASE WHEN p THEN 1 ELSE 0 END)` (or `COUNT(*) FILTER` — libSQL supports it, but the codebase prefers the `SUM(CASE …)` idiom) |
> | `DO $$ … to_regclass(…) … $$` guards in migrations | plain `CREATE INDEX IF NOT EXISTS … ON …(…) WHERE …;` (SQLite supports `IF NOT EXISTS` + partial indexes natively) |
> | Dates: comparing `started_at::timestamptz` | compare ISO-8601 **strings** directly (the app only ever persists `toISOString()`); guard malformed values with the `GLOB` date-prefix expr |
>
> **Canonical pattern to mirror** (already SQLite-correct, verified against the live store): `buildUsageFilterClause`, `aggregateSqliteUsage`, and the `SESSION_STARTED_AT_TS_EXPR` / `SESSION_STARTED_AT_BOUNDS_EXPR` date guards in `sqlite.ts`. Before finishing any SQL change, grep your diff for `ILIKE`, `::`, `EXTRACT`, `~ '`, `to_regclass`, `DO $$`, `timestamptz` — there must be **zero** matches in new SQL.
>
> **Analytics invariant (FEA-2038):** dashboard analytics/usage/branch reads must compute **metadata-only SQL aggregates** (`COUNT`/`SUM`/`GROUP BY` over the metadata/token tables) and must **never hydrate the full session/event corpus into JS** — doing so OOMs the db-host utilityProcess (exit code 5) on a large local history. Row-loading list reads default to a 90-day window (all-time via an explicit toggle); aggregates stay all-time (cheap in SQL).
>
> **Migrations** are sequential `000N_<name>/migration.sql`, hand-authored in SQLite dialect (see `0001_init`, `0002_idx_token_events_session_created`, `0003_idx_branch_name`). Use the next unused contiguous prefix. The already-merged FEA-2030 timestamped migration is immutable and grandfathered by the manifest order; do not add more timestamp-prefixed migrations. Regenerate the gitignored manifest with `node apps/desktop/scripts/generate-migrations-manifest.mjs` after adding one. (This supersedes the timestamp-prefix and Prisma-Postgres-generation guidance below.)

The local store schema is owned by `apps/desktop/prisma/schema.prisma` (independent of the cloud schema in `packages/database` by design — FEA-1736 decision 6). The generated client output `src/main/database/generated/` is gitignored; `pnpm db:generate` produces it and runs automatically inside `build`, `typecheck`, and `test`.

Schema changes are forward-only Prisma migrations applied at boot by the runtime migration runner (`src/main/database/migration-runner.ts`, PLN-886 Phase 2). To change the schema: edit `prisma/schema.prisma`, generate a migration (`pnpm db:migrate:dev` against a throwaway Postgres via `pnpm db:dev`), and commit it under `prisma/migrations/`. The runner records applied migrations in a `_desktop_migrations` tracking table, applies each pending migration in its own transaction (no down-migrations), and refuses to boot on checksum drift or a downgraded app — surfaced as an Agent Monitor boot failure with DB IPC disabled, never a crash loop. A prebuild step (`scripts/generate-migrations-manifest.mjs`) embeds the migrations into the gitignored `src/main/database/migrations-manifest.ts` so they survive asar packaging (mirrors `build-info.ts`).

**A migration is immutable once it merges to `main`.** Editing, deleting, or renaming a committed `migration.sql` changes its embedded checksum, so the runner refuses to boot for any store that already applied the old bytes (the exact checksum-drift refusal above). Never edit a merged migration; add a NEW forward migration instead. CI enforces this for both Prisma targets (cloud `packages/database` and desktop `apps/desktop`) via the `migration-immutability` job in `.github/workflows/pr-test.yml`, gated on the `database` and `desktop_database` path filters. If a local dev store hits a checksum-drift refusal after pulling `main` (because it applied an interim version of a migration that was corrected before any desktop release shipped it), reset the local store to recover: quit the app and delete the `agent-dashboard.sqlite*` files under Electron's `userData` path (macOS: `~/Library/Application Support/Closedloop/agent-dashboard.sqlite*`, including the `-wal`/`-shm` sidecars); it rebuilds from scratch on next boot. Only do this for unreleased migration churn; a released migration is permanent and must be fixed forward, never by editing it.

New desktop migration directory names must use the next contiguous `000N_` prefix in `apps/desktop/prisma/migrations/`. Do not rename the existing timestamp-prefixed FEA-2030 migration; it has a pinned manifest sort key because merged migrations are immutable. Do not insert a pending migration before an already-applied migration: changing manifest order can create a `HistoryGap` refusal on upgraded installs.

`src/main/database/baseline-schema.ts` holds the FROZEN legacy DDL snapshot as of the runner cutover plus the `BASELINE_MIGRATIONS` it represents. **Do not edit it or append to `BASELINE_MIGRATIONS`** — it exists only to baseline a pre-runner install (re-assert the idempotent legacy DDL once, then record the baseline migrations as applied *without executing* them). Post-cutover schema lives only in new migrations.

The migration-refusal contract (`MigrationRefusalKind`, `DesktopMigrationError`, `userFacingMigrationRefusal`) lives in `src/main/migration-refusal.ts`, **deliberately outside `src/main/database/`**. Boot files (`app.ts` surfaces a refusal as an Agent Monitor boot failure) cannot statically import the database runtime — the boundary guard (`test/agent-dashboard-boundary.test.ts`) forbids any boot-graph value import into `src/main/database/**` (which transitively pulls in SQLite). Keeping the pure, DB-free contract in its own module lets the runner (producer) and the boot handler (consumer) share it without dragging the DB runtime into the boot bundle.

### Typed Prisma client (PLN-886 Phase 3)

`src/main/database/prisma-client.ts` (`createDesktopPrisma`) wires `@prisma/adapter-libsql` onto the ALREADY-OPEN SQLite handle — one SQLite instance, one Prisma client, created inside `openSqliteAgentDatabase` and exposed as `agentDatabase.prisma`. Because the client shares SQLite's single physical connection with the raw `storeDb` path, **every Prisma write and `$transaction` must go through `prisma.write(fn)`**, which serializes through the same write queue the raw path uses. `prisma.client` is typed as a **read-only facade** (`DesktopPrismaReader`) — calling a mutation method on it is a compile error, so the write-queue rule is type-enforced, not convention. The community adapter is exact-version-pinned and confined to this one module so a swap to the FEA-1736 in-house `SqlDriverAdapter` fallback touches one file — **production code must not import the adapter anywhere else**. The sole exception is the Phase 0 characterization test (`test/prisma-sqlite-adapter.test.ts`), which imports the adapter directly on purpose to exercise the raw client↔adapter contract against the real schema; all other tests go through the factory (`test/prisma-test-utils.ts`).

Adoption is incremental (NOT big-bang), one module per PR. **End state: ONE client.** Queries that don't map cleanly to a typed delegate are NOT kept on a separate raw `storeDb` handle — they run on the same `DesktopPrisma` client via Prisma's raw escape hatch (`$queryRawUnsafe` for reads, `$executeRawUnsafe` for writes), and the standalone `storeDb` accessor + `DbClient` types + `withStoreDb` wrapper are deleted in the final cleanup PR. The community adapter implements `queryRaw`/`executeRaw`, so raw SQL works through it (characterized in the Phase 0 `test/prisma-sqlite-adapter.test.ts`: `$queryRaw`/`$transaction` and the `$queryRawUnsafe`/`$executeRawUnsafe` positional-param variants the write path uses). Raw **writes** are reachable only inside `prisma.write(...)` (the full client); raw **reads** outside a write use `$queryRaw`/`$queryRawUnsafe`, which the `DesktopPrismaReader` facade re-exposes read-only (`$executeRaw*`/`$transaction`/typed mutations stay compile-errors on the facade). The read-vs-write split on `$queryRaw*` is by convention — SQL can't be statically classified — so never smuggle a mutation through it; raw writes go through `prisma.write`.

### Collection-mode mutual exclusivity (FEA-1839)

A harness's live telemetry is captured by exactly one channel at a time: the Closedloop hook handler (`"hooks"`) or the live JSONL file watcher (`"watcher"`) — never both. Running both for one harness double-counts every tool-call and lifecycle row. `getActiveCollectionMode(harness, hooks)` in `src/main/collectors/collection-mode.ts` is the single source of truth for this routing; `CollectorManager` starts a live watcher only for `"watcher"`-mode harnesses (`"hooks"`/`"disabled"` run the idempotent boot import once, no watcher). Every new harness must derive its live-collection decision from this function — do not add an inline hooks-installed conditional at a call site. The `mutual-exclusivity-monitor` records a single `mutual_exclusivity_violation` event row if both channels ever emit the same harness session in one process lifetime.

## Idempotent Re-Processing: Revision Constants

Desktop uses three independent version constants to trigger idempotent re-derivation of local data. Each governs a different concern — bumping one does NOT affect the others.

### DATA_REVISION (`src/main/collectors/data-revision.ts`)

**Current value:** 3. Controls session-level re-derivation from source transcripts.

Every session row is stamped with `data_revision = DATA_REVISION` at import time. On boot, `data-revision-rebuild.ts` queries `SELECT ... FROM sessions WHERE data_revision != <current>`, re-parses each stale session from its source transcript, and transactionally replaces derived rows (events, agents, token_usage, session_artifact_links).

**When to bump:** Parser or import semantics changed in a way that should re-derive already-imported sessions. Examples: event ID generation changed (rev 3), dedup/merge logic changed (rev 2), new fields extracted from transcripts.

**How:** Increment the number in `data-revision.ts`, add a changelog entry to the revision history comment. Sessions with the old value are rebuilt on next app launch; active sessions heal via their normal reimport path. Sessions whose source transcript is gone are left untouched.

**Does NOT trigger:** enrichment re-run or catchup cache invalidation.

### EXTRACTOR_VERSION (`src/main/collectors/artifact-ref-extractor.ts`)

**Current value:** 5. Controls artifact-reference re-extraction from parsed sessions.

The artifact-ref-extractor runs at parse/import time and produces `session_artifact_links` rows. Each link row is stamped with `extractor_version`. The backfill pass (`artifact-link-backfill.ts`) queries for sessions whose link rows carry a stale extractor version and re-runs the extractor.

**When to bump:** Extraction semantics changed — new artifact kinds detected, relation classification changed (e.g. commit refs changed from `output` to `created`), new regex patterns added, confidence scoring changed.

**How:** Increment `EXTRACTOR_VERSION` in `artifact-ref-extractor.ts`. Existing installs will re-derive artifact links on next boot via the backfill pass. Enrichment of newly-created artifacts is triggered by the enrichment runner's post-import sweep.

**Does NOT trigger:** session re-parse (that's DATA_REVISION) or catchup cache invalidation.

### PERSIST_VERSION (`src/main/collectors/catchup-cache.ts`)

**Current value:** 2. Controls the file-level catchup cache (mtime/size fingerprints).

The catchup cache tracks which transcript files have been imported. When `PERSIST_VERSION` changes, the entire cache is invalidated, forcing a one-time full re-scan of all transcript files on next boot.

**When to bump:** Cache format or fingerprint semantics changed (e.g. new fields in the cache entry, different hashing). NOT needed for parse output changes (use DATA_REVISION for that).

**How:** Increment `PERSIST_VERSION` in `catchup-cache.ts`. Next boot discards the cache and re-scans all files.

### Enrichment State Machine (`src/main/enrichment/`)

Enrichment is pull-based, NOT revision-driven. Artifacts have an `enrichment_state` column with four values:

| State | Meaning |
|-------|---------|
| `null` | Never enriched. Sweep will attempt enrichment. |
| `provisional` | Enriched but may change (e.g. open PR, active branch). Sweep will re-check. |
| `final` | Enriched and stable (merged PR, validated commit). Sweep skips. |
| `not_applicable` | Cannot be enriched (no git dir, gh unavailable, max failed attempts). Sweep skips. |

**Triggers:** Three events cause an enrichment sweep: post-import, app startup, app focus. The sweep queries for artifacts with `enrichment_state IS NULL OR enrichment_state = 'provisional'` and processes them.

**To force re-enrichment of all artifacts:** Reset `enrichment_state` to `NULL` on the target rows:
```sql
UPDATE artifacts SET enrichment_state = NULL, enrichment_attempts = 0 WHERE ...;
```
The next sweep trigger will re-enrich them. There is no `ENRICHMENT_REVISION` constant — enrichment is stateless and re-derives from git/gh on each attempt.

**Constants:** `MAX_ENRICHMENT_ATTEMPTS = 5` (failed attempts before `not_applicable`), `ENRICHMENT_SWEEP_DEBOUNCE_MS = 5min`, `LEASE_STALE_MS = 5min`, `GH_RATE_LIMIT_INTERVAL_MS = 6s`.

### Summary: Which Constant To Bump

| Change | Bump |
|--------|------|
| Parser extracts different data from transcripts | `DATA_REVISION` |
| Artifact-ref extraction logic changed | `EXTRACTOR_VERSION` |
| Catchup cache format/fingerprint changed | `PERSIST_VERSION` |
| Enrichment logic changed (git/gh queries) | Reset `enrichment_state` to NULL |
