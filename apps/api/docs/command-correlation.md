## Minting Point

`commandId` is minted in exactly one place: the POST handler in [`/apps/api/app/compute-targets/[id]/commands/route.ts`](/apps/api/app/compute-targets/[id]/commands/route.ts).

The minting sequence is:

1. `const requestId = crypto.randomUUID();` — mints the HTTP-request trace ID
2. `desktopCommandStore.createCommand(target.id, rewrittenInput, buildTelemetryTraceContext({ computeTargetId, operationId, requestId }))` in [`/apps/api/lib/desktop-command-store.ts`](/apps/api/lib/desktop-command-store.ts) — persists the command row; `commandId` is generated by Prisma's `@default(uuid(7))` on the `DesktopCommand.id` column (`packages/database/prisma/schema.prisma`). Note that the third argument is a fresh `buildTelemetryTraceContext({ computeTargetId, operationId, requestId })` call built inline at the call site — it intentionally does NOT include `commandId` (which is not yet known) and is distinct from the `traceContext` variable (re)built after `createResult` returns and used for lifecycle-event emission downstream.
3. `const { commandId } = createResult.command;` — the caller extracts `commandId` from the returned record

**Invariant:** `commandId` is NEVER minted elsewhere. All other files receive and propagate an existing `commandId`.

**Security requirement:** `commandId` MUST be generated using a cryptographically secure pseudo-random number generator (CSPRNG). The current implementation uses Prisma's `@default(uuid(7))` on the `DesktopCommand.id` column (`packages/database/prisma/schema.prisma`), which generates UUID v7 values with a 48-bit timestamp prefix and ~74 random bits per value. Sequential IDs (e.g., auto-increment integers) and schemes with low unpredictability such as UUIDv1 (which embeds the MAC address) are **prohibited**; UUID v7 is acceptable because its ~74 random bits make individual values unguessable despite the timestamp prefix. `commandId` appears in the events endpoint URL (`/compute-targets/:id/commands/:commandId/events`) and must be unguessable to prevent unauthorized clients from polling or injecting events for a command they did not create.

## Propagation Path

`commandId` travels through three distinct legs after minting.

### Leg 1 — ECS-relay dispatch (HTTP)

After `commandId` is minted, the route calls `toRelayOperation(commandId, rewrittenInput)` (defined in [`/apps/api/app/compute-targets/relay-command-helpers.ts`](/apps/api/app/compute-targets/relay-command-helpers.ts)), which embeds `commandId` in `params.commandId` of a `RelayOperationDispatchRequest`. The route then calls `dispatchToRelay()` (local helper in the same route file), which:

1. calls `toWireCommandFromRelayOperation()` to convert to `WireCommandPayload`
2. calls `withCorrelationContext(wireCommand, { requestId, computeTargetId: targetId })` (defined in [`/apps/api/app/compute-targets/relay-command-helpers.ts`](/apps/api/app/compute-targets/relay-command-helpers.ts)) — returns a new `WithCorrelation<WireCommandPayload>` with `requestId` and `computeTargetId` set; `gatewaySessionId` is NOT set at this stage
3. calls `toEnvelope(correlatedCommand)` (defined in [`/apps/api/lib/desktop-gateway-wire.ts`](/apps/api/lib/desktop-gateway-wire.ts)) — wraps with `protocolVersion`, `messageId`, `timestamp`
4. `fetch`es `${RELAY_API_URL}/dispatch` via HTTP POST with `Content-Type: application/json`, `x-internal-secret: <secret>` header, `AbortSignal.timeout(5000)` (5-second timeout), body `{ targetId, operation: <envelopedCommand> }`

[`/apps/relay/src/index.ts`](/apps/relay/src/index.ts) receives the dispatch, looks up the connected desktop worker by `targetId`, and — if the worker is connected — emits a `"desktop.command"` socket.io event with the full wire envelope (containing `commandId`). If no worker is connected, the relay returns `200 OK` with body `{ delivered: false, reason: "target_not_connected" }`; the successful delivery path returns `200 OK` with body `{ delivered: true }`.

**Dispatch-result handling in `dispatchToRelay()`** (`apps/api/app/compute-targets/[id]/commands/route.ts`):

- `response.ok` is treated as success regardless of the `delivered` field — `dispatchToRelay()` returns `true` and the caller emits `CommandDispatched` telemetry even when the relay signaled `delivered: false`. See [GAP-004](#gaps).
- A JSON-parse failure on a 2xx body is **fail-open**: `response.json().catch(() => ({ delivered: true }))`. If the relay ever returns a 2xx response with an empty body, HTML, or malformed JSON, the API treats it as a successful delivery and logs `delivered: true`. This behavior is intentional today (keeps command lifecycle moving on transient relay glitches) but should be understood when interpreting `CommandDispatched` telemetry. See [GAP-004](#gaps).

### Leg 2 — In-process EventBus dispatch (fallback when `RELAY_API_URL` is unset)

The route calls `relayEventBus.publishOperation(target.id, relayOperation)` (defined in [`/apps/api/lib/relay-event-bus.ts`](/apps/api/lib/relay-event-bus.ts)), passing a `RelayOperationDispatchRequest`.

**Note:** Correlation fields (`requestId`, `gatewaySessionId`, `computeTargetId`) are NOT attached on the in-process path. This is a material behavioral asymmetry between the two dispatch modes.

### Leg 3 — Event ingestion (desktop → API)

Desktop executes the command and streams result/error/done events back through the gateway WebSocket. [`/apps/api/lib/desktop-gateway-socket-server.ts`](/apps/api/lib/desktop-gateway-socket-server.ts) receives these events and calls `desktopCommandStore.ingestCommandEvent(commandId, event)` (in [`/apps/api/lib/desktop-command-store.ts`](/apps/api/lib/desktop-command-store.ts)), which updates the event sequence and notifies subscribers. [`/apps/api/lib/desktop-relay-event-bridge.ts`](/apps/api/lib/desktop-relay-event-bridge.ts) bridges events to the relay event bus via `operationId` lookup.

Note: [`/apps/api/app/compute-targets/[id]/commands/[commandId]/events/route.ts`](/apps/api/app/compute-targets/[id]/commands/[commandId]/events/route.ts) is the SSE read path (API → browser), not the event write path.

## Correlation Context Contract

This section addresses the PRD §9 "header contract" requirement by documenting how correlation actually travels today.

### Type definition

The `WithCorrelation<T>` generic type is defined in [`/apps/api/lib/desktop-gateway-types.ts`](/apps/api/lib/desktop-gateway-types.ts). It intersects `T` with three optional envelope-level routing fields: `requestId` (HTTP-request trace ID), `gatewaySessionId` (desktop gateway WebSocket session — treat as session-token sensitive, never log or expose in client-facing responses), and `computeTargetId` (identifies the target compute node). `WireCommandPayload` (in the same file) already carries `commandId` as a first-class required field; `WithCorrelation<T>` adds the three envelope-level routing fields on top.

### ECS-relay wire format

The HTTP POST to `${RELAY_API_URL}/dispatch` has the following shape:

| Property | Value |
|---|---|
| Method | `POST` |
| URL | `${RELAY_API_URL}/dispatch` |
| Headers | `Content-Type: application/json`, `x-internal-secret: <INTERNAL_API_SECRET>` |
| Timeout | `AbortSignal.timeout(5000)` (5 seconds) |
| Body | `{ targetId: string, operation: Envelope<WithCorrelation<WireCommandPayload>> }` |

### Field population at minting time

`requestId` and `computeTargetId` are set by `dispatchToRelay()` at minting time. `gatewaySessionId` is declared in `WithCorrelation<T>` but is NOT populated by the minting route; it is reserved for population by the socket server or desktop session path (behavior not yet fully defined — see GAP-002).

### No HTTP header named `x-desktop-command-id` exists

The PRD §9 "header contract" naming is aspirational. In the current implementation, correlation fields travel as JSON body fields in the wire envelope, not as HTTP headers. A repo-wide grep returns zero matches for `x-desktop-command-id`. **PR reviewers must explicitly acknowledge this reconciliation during review** to prevent AC-001 rejection on a technicality.

### Error behavior

All three correlation fields are optional in the `WithCorrelation<T>` type. No validation is performed on the relay `/dispatch` handler or the event ingestion side. A missing field degrades traceability but does not fail or reject the command. This is an open policy question, not an intentional omission — see GAP-002.

**Security note:** `requestId`, `gatewaySessionId`, and `computeTargetId` are internal correlation identifiers. They MUST NOT be echoed in HTTP error responses returned to API callers — doing so would expose internal routing topology to untrusted clients.

## Cardinality Discipline

### The rule

`commandId`, `requestId`, `operationId`, and similar high-cardinality correlation handles MUST appear only in structured log body entries and NEVER be elevated as a Datadog metric tag. The exclusion rationale varies by field:

- **Cardinality only:** `commandId`, `operationId`, `requestId`, `loopId`, `loopSessionId`, `jobId` — these are excluded solely because they produce unbounded tag cardinality in Datadog, which inflates metric costs.
- **Cardinality + privacy:** `userId`, `machineName` — these are excluded for *both* cardinality AND privacy reasons. Even if Datadog cardinality limits were removed, PII-adjacent identifiers must not appear as metric dimensions that could be queried or exported outside the log pipeline.

Always identify which category applies before proposing a new field for the allowlist.

### Allowed metric tags

The fields accepted by the metric types in [`/packages/observability/telemetry/metrics.ts`](/packages/observability/telemetry/metrics.ts) differ between the two emitters:

- **`QueueMetric`** (passed to `emitQueueMetric()`): `computeTargetId`, `fromStatus`, `toStatus`, `commandId` — plus the base fields `metric`, `count`, `value`, `timestamp`.
- **`ProtocolMetric`** (passed to `emitProtocolMetric()`): `computeTargetId`, `gatewaySessionId` — plus the base fields `metric`, `count`, `value`, `timestamp`.

The conceptual metric-tag allowlist in [`packages/observability/README.md#metric-tag-allowlist`](../../../packages/observability/README.md#metric-tag-allowlist) additionally names `errorClass`, `origin`, and `state`. Those tags are **not** part of the type-enforced surface of `QueueMetric` or `ProtocolMetric` today — they apply to broader metric emissions (e.g., `emitValidationFailedCounter()` and ad-hoc `log.*` metric markers) and are governed by convention and code review, not by the two emitter type signatures.

### How enforcement works

`QueueMetric` and `ProtocolMetric` bound which fields `emitQueueMetric()` and `emitProtocolMetric()` accept at the type level. Two nuances matter:

1. `commandId` is an optional field on `QueueMetric` (it is used as a structured log body field on command-lifecycle transitions). TypeScript will therefore accept `commandId` in an `emitQueueMetric()` call — the cardinality rule excluding `commandId` from metric tag emission is enforced by **convention and code review**, not by a compile-time guarantee.
2. `errorClass`, `origin`, and `state` are not present on either `QueueMetric` or `ProtocolMetric`. Passing them directly to `emitQueueMetric()` or `emitProtocolMetric()` is a TypeScript error; they appear in the README allowlist as permitted tags for the wider universe of metric-bearing log emissions, not for these two emitters specifically.

See [`/packages/observability/telemetry/metrics.ts`](/packages/observability/telemetry/metrics.ts) for the current type definitions.

See also: [metric-tag allowlist](../../../packages/observability/README.md#metric-tag-allowlist) in the observability README.

## Gaps

Known mismatches between the PRD/plan and the current implementation, kept here so readers of the "authoritative" doc are not misled about dispatch semantics.

- **GAP-001 — PRD header-naming aspiration.** PRD §9 names an `x-desktop-command-id` HTTP header, but correlation actually travels as JSON body fields on `WireCommandPayload` via `WithCorrelation<T>`. See the ["No HTTP header named `x-desktop-command-id` exists"](#no-http-header-named-x-desktop-command-id-exists) section.
- **GAP-002 — Correlation-field validation.** All three `WithCorrelation<T>` fields are optional; neither the relay `/dispatch` handler nor the event ingestion side validates them. Policy question, not an intentional omission.
- **GAP-003 — Metric-tag allowlist governance.** The README lists an allowlist but does not specify whether additions require governance review.
- **GAP-004 — Dispatch result vs. `CommandDispatched` telemetry divergence.** `dispatchToRelay()` returns `true` on any `response.ok`, so `CommandDispatched` telemetry fires even when the relay response body carried `delivered: false` (e.g., `reason: "target_not_connected"`). The `response.json().catch(() => ({ delivered: true }))` fallback additionally fail-opens on empty / non-JSON 2xx bodies. This means `CommandDispatched` telemetry is not a reliable signal of actual worker delivery. Fixing the behavior (check `result.delivered` before emitting `CommandDispatched`) and closing the API-side test gap for the `delivered: false` branch of `POST /compute-targets/:id/commands` are tracked as a follow-up Closedloop feature.
