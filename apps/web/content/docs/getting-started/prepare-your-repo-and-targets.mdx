---
title: Prepare Your Repo and Targets
description: Make local execution predictable before you scale up docs or code generation.
---

Most first-loop failures are not AI failures. They are environment failures: wrong PATH, a missing `gh` auth, an under-specified artifact, or a sandbox that excludes the repo. Fix these three checklists before you start issuing loops.

## Repository checklist

- the default branch is stable and up to date
- contributor permissions are clear
- there is a clean branch-naming strategy for loop output (ClosedLoop uses `closedloop/<ticket>-<slug>` by default)
- core project commands are documented in `CLAUDE.md` or `AGENTS.md` so agents can find them
- the repo can be cloned by `gh repo clone <org/repo>` without interactive prompts
- tests, lint, and typecheck can each be run with a single command

:::note
If your project does not already have a `CLAUDE.md`, consider running `/agent-bootstrap` from a checkout inside the sandbox. It produces a set of project-specific expert agents plus a `critic-gates.json` that tells the code plugin which critics to run for each module.
:::

## Compute target checklist

- the desktop client is installed and its three Dashboard cards are green
- required CLIs (`claude`, `gh`, `codex`, `python3`, `git`) are detected under **Settings → CLI Tools**
- the target reports its provider support accurately in the web app's Runtime picker
- the sandbox base directory contains (or can contain) a clone of the repo you plan to execute against
- network access is sufficient to reach `relay.closedloop.ai`, `api.closedloop.ai`, and GitHub

## Artifact checklist

- the artifact has a clear title and a single-paragraph goal statement
- the scope is concrete (what is in, what is out)
- **acceptance criteria** are listed and each one is independently reviewable
- context is attached where needed (linked PRDs, existing designs, related tickets)
- the artifact passes the four PRD judges (auditor, dependency, scope, testability) when evaluated in plan mode

When these three layers are ready, loops are much less likely to fail for environmental reasons.

## Multi-repo setups

If your work touches more than one repository (for example, a backend and a frontend that ship together), prepare the peer repos the same way you prepare the primary:

- clone them into your sandbox
- ensure each has a `.closedloop-ai/.repo-identity.json` with a stable `name`
- verify each can be discovered from the primary via `discover-repos.sh` or by passing `--add-dir <path>`

See [Multi-repo development](../mechanisms/multi-repo) for the full mechanism.

## What success looks like

A well-prepared project produces its first loop end-to-end without any approvals hanging, without a single missing-binary telemetry event, and with a judge report that cleanly scores against a PRD that was reviewable before the loop started.
