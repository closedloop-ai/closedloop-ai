---
title: PRD to plan to execution to PR
description: The core delivery workflow that turns defined intent into reviewable output.
---

This is the canonical ClosedLoop.ai workflow. Everything else is a variation on it.

## Input

- a PRD or feature request with clear scope, constraints, and success criteria
- a repository with a clean default branch
- a compute target that reports `ready`

## Steps

### 1. Create or refine the PRD

Use the [PRD template](../resources/prd-template) or `@code:prd-creator` skill. Cover at least: problem statement, desired outcome, constraints, success criteria, risks.

:::tip
Run the PRD through `@judges:run-judges --artifact-type prd` to catch gaps before you spend a loop on a weak spec.
:::

### 2. Generate the plan

```bash
claude /code:code --prd prd.md
```

The `code` plugin:

- pre-explores the codebase, writing `code-map.json` and `requirements-extract.json`
- drafts `plan.json` and `plan.md` via `plan-draft-writer`
- runs `PLAN_VALIDATION_SEQUENCE` until the plan passes validation

The orchestrator stops at the plan review checkpoint with `AWAITING_USER` in `state.json`.

### 3. Review and amend the plan

Open `plan.md`. Review:

- does it cover the scope?
- are tasks sized appropriately (S/M/L)?
- are acceptance criteria explicit?
- are cross-repo dependencies surfaced?

If you need changes, use:

```bash
claude /code:amend-plan --workdir <path> --message "<text>"
```

The amend extractor classifies your input as a directive, a question, a confirmation, or unstructured feedback, and updates `plan.json` accordingly.

### 4. Let critics run

Relaunch the loop. The orchestrator:

- evaluates the plan in simple mode (`plan-evaluator` via `eval-cache`)
- runs critics in parallel (gated by `critic-cache`)
- merges critic feedback into `plan-writer` merge mode
- finalizes the plan with acceptance criteria

Critic reviews land in `reviews/*.review.json`.

### 5. Execute implementation

The loop enters Phase 3. For each task:

1. `@code:verification-subagent` checks whether the task is already done (`VERIFIED` or `NOT_IMPLEMENTED`).
2. `@code:implementation-subagent` implements the task with self-verification gates.
3. A haiku marks the task `[x]` in the plan.

Each implementation agent has a 4-iteration loop budget and must emit `<promise>IMPLEMENTATION_VERIFIED</promise>`.

### 6. Validate

- Phase 4: code simplification pass.
- Phase 5: testing and build validation, max 20 attempts.
- Phase 6: visual QA via Playwright if `visual-requirements.md` is non-empty.

### 7. Grade with judges

The `judges` plugin runs plan and code judges in batches, producing `plan-judges.json` and `code-judges.json`.

### 8. Ship

The loop commits, pushes a branch, and creates a PR via `git_pr` operations. You review the PR, merge, and move on.

## Output

- a tracked execution record (`state.json`, `log.md`, full session work directory)
- review context (plan, critics, judges, learnings)
- a concrete PR with a diff that matches the plan

## When to deviate

- For very small changes (doc edits, single-test additions), consider skipping critics and judges; run a minimal loop.
- For large features, decompose into multiple PRDs and run them in parallel. See [Parallel feature development](./parallel-feature-development).
- For cross-repo features, use the cross-repo discovery phase to generate peer-specific PRDs.

## What you gain

Every step has an artifact. Every phase has a gate. Every output has a score. The chain from PRD to PR is observable end-to-end, and you can reproduce, judge, and learn from every run.
